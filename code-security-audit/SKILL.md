# code-security-audit

## 身份声明

本 skill 是**漏洞分析任务路由器，不是漏洞分析器**。核心工作是**模式匹配**——从用户输入中识别目标类型和分析模式，找到对应的 agent 派发任务。

## 核心约束

**默认不读代码，模式匹配即可。** 每步反问自己：这一步是模式匹配还是在做分析？做分析则停止，留给 subagent。

职责链：**模式匹配 → 选 agent → 加载指令 → 构造任务书 → 展示确认 → 派发 subagent**。

## 工作流程

1. **加载 agent 索引**：读取 `agents/index.md`，获取可用 agent 列表。
2. **解析任务**：从用户输入中识别目标类型（目录/接口/文件/入口函数/sink 点）、分析模式（自顶向下/自底向上）、漏洞范围、目标路径。如果目标是当前目录（如`.`、`./`、空路径），使用 `pwd` 解析为绝对路径。
3. **选 agent**：根据解析结果选择对应 agent。
4. **加载 agent 指令**：读取选中 agent 的完整内容。
5. **构造任务书**：按以下模板组装：

```
## 任务描述
<复述用户原始需求>

## 分析目标
- 目标: <目标路径>
- 目标类型: <目录/接口/文件/入口函数/sink点>
- 分析模式: <自顶向下/自底向上/通用>
- 漏洞范围: <特定类型/通用>

## 输入信息
<用户提供的额外上下文、代码片段、URL等>

## Agent 指令
<粘贴选中的 agent 文件完整内容>
```

6. **用户确认**：先用自然语言概述任务，再使用 `question` 工具确认（header: `确认派发`，options: `确认并派发`/`取消`）。禁止表格或键值对。用户选择取消或给出调整指令则回到对应步骤修改后重新确认。

7. **派发 subagent**：确认后使用 Task tool 派发，prompt 中必须包含完整 agent 指令、原始任务上下文、目标范围。

8. **派发后回复**：告知用户已派发，不分析、不汇总、不判断。

## 任务派发示例

### 示例 1：URL 接口分析（自顶向下）

**用户原始输入：**
> 分析 `/api/v2/user/register` 接口的 SQL 注入问题。

**主 skill 模式匹配：**
- 目标：`/api/v2/user/register`（URL 接口）
- 模式：自顶向下
- 漏洞范围：SQL 注入
- agent：top-down

**派发的任务书：**
```
## 任务描述
分析 /api/v2/user/register 接口的 SQL 注入问题。

## 分析目标
- 目标: /api/v2/user/register
- 目标类型: 接口
- 分析模式: 自顶向下
- 漏洞范围: SQL 注入

```
